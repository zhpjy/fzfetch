# Fzfetch 前后端闭环联通设计

## 1. 目标

在不重构现有前端目录的前提下，打通 Fzfetch 的完整前后端交互链条，使当前 React + Vite 前端与 Rust + Axum 后端形成稳定闭环，满足以下结果：

- 前端通过 WebSocket 与后端实时搜索接口联通。
- 用户快速输入时，前端和后端都只接受最新请求结果，旧结果不会污染界面。
- 后端索引后台刷新完成后，前端收到 `INDEX_REFRESHED` 广播并按当前查询词静默重查。
- 下载幽灵文件时，后端返回 `410 Gone`，前端静默移除结果项并给出轻提示。
- 前端显式展示连接状态、搜索状态、后台刷新状态、无结果状态与错误状态。
- 移除“手动刷新”能力，完全遵循既定架构中的自动刷新机制。

## 2. 非目标

这次工作不包含以下内容：

- 不按新的 feature-first 路径重构整个 `frontend/src` 目录。
- 不新增“主动刷新索引”的 HTTP 或 WebSocket 接口。
- 不把模糊匹配、索引缓存或文件扫描逻辑挪到前端。
- 不改动后端的核心索引生命周期设计，只补足前端联调所需的最小后端支撑。

## 3. 当前现状

### 3.1 后端

后端已经具备以下能力：

- `/ws` 支持 `{"req_id","query"}` 搜索请求。
- 返回 `{"req_id","data":[...]}` 搜索结果。
- 刷新完成后广播 `{"type":"INDEX_REFRESHED"}`。
- `/download?path=...` 对幽灵文件返回 `410 Gone`。
- 懒加载 `cache.txt`、TTL 触发刷新、空闲卸载内存索引等机制已经存在。

### 3.2 前端

前端已经有基础界面和几个 hooks，但目前存在以下问题：

- `useSearchSocket` 状态过于粗糙，只区分 `ready / refreshing / disconnected`。
- “刷新”按钮与既定架构冲突，因为后端没有主动刷新接口。
- 重连、错误、后台刷新、搜索中状态没有清晰边界。
- 空查询、无结果、断连、刷新中等场景的界面语义混在一起。
- `410 Gone` 已有处理，但与搜索状态、列表选中状态的协调还不完整。
- 现有 WebSocket URL 写死到 `hostname:3000`，不适合统一走 Vite 代理。

## 4. 方案选择

采用“轻量状态收口，保留现有 UI 和目录”的方案：

- 保留 `App.tsx + hooks + components` 的现有工程组织。
- 以 `useSearchSocket` 作为搜索与连接状态的唯一来源。
- 以 `useDownload` 作为下载与幽灵文件清理的唯一来源。
- 仅做必要的后端适配与前端协议清理，不引入新的服务端接口和前端状态管理库。

选择该方案的原因：

- 它能以最小变更打通完整闭环。
- 它不会把范围扩大到目录重构。
- 它能把现在分散在组件中的协议逻辑收回到 hook 层，便于后续迭代。

## 5. 架构设计

### 5.1 WebSocket 通道

前端维持单条 `/ws` 长连接，承担三类职责：

1. 在输入后经“微防抖”发送搜索请求。
2. 接收搜索结果，并只接受与本地最新 `req_id` 对应的响应。
3. 接收 `INDEX_REFRESHED` 广播，若当前查询非空，则自动重发当前查询。

这里保持双重防护：

- 前端本地只接受最新 `req_id` 响应。
- 后端连接内 `AtomicU64` 只发送最新 `req_id` 响应。

这样即使网络抖动或搜索计算耗时变化，也不会出现旧结果把新结果覆盖的问题。

### 5.2 搜索状态模型

前端把状态拆成两个维度，而不是强行合并成单个模糊状态：

- 连接维度
  - `connecting`
  - `ready`
  - `disconnected`
  - `error`
- 工作维度
  - `idle`
  - `searching`
  - `refreshing`

设计原因：

- “后台刷新中”并不等于“连接异常”。
- “正在搜索”也不等于“正在刷新索引”。
- 只有拆分维度，界面才能准确表达“连接正常，但索引后台刷新中”的场景。

最终 `StatusIndicator` 只负责渲染，不再承担交互动作。

### 5.3 下载闭环

下载流程保持极简：

1. 前端点击结果项或按下 `Enter`。
2. 前端请求 `/download?path=...`。
3. 若返回 `410 Gone`，前端静默移除该结果项，并提示“文件已不存在”。
4. 若成功，前端触发浏览器下载。
5. 若失败，前端展示错误提示，但不清空搜索结果。

这里不在前端提前做文件存在性探测，完全遵循后端“惰性校验”架构。

### 5.4 键盘交互

保留现有键盘导航模型，但补齐边界：

- `ArrowUp / ArrowDown` 在结果列表中循环移动。
- `Enter` 只在存在合法选中项时触发下载。
- `Escape` 清空当前查询并清空结果。
- 当结果集变化或某项因 `410` 被移除时，选中项自动回落到有效范围。

### 5.5 空态与反馈

界面需要区分以下状态：

- 空查询：提示用户输入关键词。
- 有查询但无结果：提示没有匹配项。
- 正在搜索：展示轻量搜索中状态，不遮挡列表。
- 后台刷新中：状态条提示“索引后台更新中”，但已有结果仍可继续查看和下载。
- 断开连接：提示服务未连接，并自动重连。
- 下载失败：toast 错误提示。

## 6. 文件设计

### 6.1 前端文件

- `frontend/src/types.ts`
  - 收敛前后端协议类型与前端状态类型。
- `frontend/src/hooks/useSearchSocket.ts`
  - 负责连接、重连、微防抖、请求纪元、本地结果接收、刷新广播重查。
- `frontend/src/hooks/useDownload.ts`
  - 负责下载、`410` 幽灵文件移除、toast。
- `frontend/src/hooks/useKeyboardNavigation.ts`
  - 负责键盘事件、选中项边界修正、`Esc` 清空查询。
- `frontend/src/components/StatusIndicator.tsx`
  - 纯展示组件，只负责把连接状态和工作状态可视化。
- `frontend/src/App.tsx`
  - 负责组装页面，不直接管理底层协议细节。
- `frontend/vite.config.ts`
  - 保持 `/ws` 和 `/download` 代理，前端使用相对路径。

### 6.2 后端文件

后端只做最小必要补足：

- `src/main.rs`
  - 若要支持前端静态资源同源部署，可在后续接入静态文件服务；本次不是必须项。
- `src/ws.rs`
  - 保持现有协议，不新增手动刷新消息。
- `src/api.rs`
  - 保持 `410 Gone` 行为不变。

## 7. 数据流

### 7.1 搜索数据流

1. 用户在输入框输入。
2. 前端微防抖后生成新 `req_id`，发给 `/ws`。
3. 后端执行搜索，只对最新 `req_id` 返回结果。
4. 前端收到结果后再次比对本地最新 `req_id`。
5. 命中才更新列表，否则丢弃。

### 7.2 刷新广播数据流

1. 后端因首次加载空缓存或 TTL 过期而执行后台刷新。
2. 刷新完成后广播 `{"type":"INDEX_REFRESHED"}`。
3. 前端若当前 query 非空，则自动按当前 query 重新发送搜索。
4. 前端状态短暂显示 `refreshing`，随后回到 `ready + idle/searching` 的组合状态。

### 7.3 下载数据流

1. 用户点击结果或按 `Enter`。
2. 前端请求 `/download?path=...`。
3. 若返回 `410`，该结果项从当前列表删除。
4. 若返回成功流，浏览器开始下载。

## 8. 错误处理

### 8.1 WebSocket 错误

- 连接建立失败或异常关闭时，前端状态设为 `disconnected`。
- 前端用固定退避重连，避免疯狂重试。
- 重连成功后状态恢复为 `ready`。

### 8.2 消息解析错误

- 非法消息只记录到控制台，不使前端崩溃。
- 当前列表与查询保持不变。

### 8.3 下载错误

- `410`：静默移除目标项，给轻提示。
- `403 / 404 / 500`：给错误提示，不移除当前项。

- `npm run build`

  - 输入关键词能拿到搜索结果。
  - 快速输入时只渲染最新结果。
  - 后端刷新后会自动重查当前查询。
  - 下载幽灵文件后结果项被移除。
  - 断开后端再恢复时前端能自动重连。

### 9.2 后端验证

- 保持已有 `cargo test` 通过。
- 若需要联调确认，可本地同时启动前后端：
  - 后端监听 `127.0.0.1:3000`
  - 前端通过 Vite 代理访问 `/ws` 和 `/download`

## 10. 风险与约束

- 当前前端没有测试框架，本次以构建验证和手工联调为主。
- 由于不重构目录，部分前端文件仍然偏大，但这次不会顺手扩大范围。
- 若后续要做生产同源部署，仍需决定是由 Axum 托管静态前端，还是前后端独立部署。

## 11. 实施边界结论

这次实现的完成标准是：

- 前端真实联通后端 `/ws` 和 `/download`。
- 搜索、刷新广播、下载、幽灵文件清理、状态展示全部形成完整闭环。
- 不引入主动刷新接口。
- 不做前端目录重构。
