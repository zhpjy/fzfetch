# Results List Overflow Design

## Goal

在窄宽度下优化搜索结果列表的可读性，减少长路径对主列表的挤压，同时消除“单击查看”和“单击下载”之间的交互冲突。

## Scope

- 调整前端结果列表的视觉层级和交互定义
- 取消“点击整行立即下载”
- 保留并强化键盘 `Enter` 下载和明确的下载按钮
- 改进长路径显示策略，避免依赖跑马灯
- 让当前选中项直接在原有两行内容内展示完整信息

不在本次范围内：

- 后端搜索协议改造
- 返回 `basename` / `display_path` 等新字段
- 新增复杂动画或持续滚动文本

## Problems

当前结果项同时承担了三个目标：

1. 展示文件名和完整路径
2. 支持点击整行下载
3. 在窄宽度下尽量保留可读性

这三者互相冲突：

- 长路径会压缩文件名，使主信息难以快速扫读
- 点击整行下载导致误触成本高
- 一旦要在点击或选中态展开更多信息，就会和下载行为打架

## Design

### 1. Interaction Model

结果列表改为“浏览/选中”和“下载”解耦：

- 单击结果行：只负责选中当前项
- 键盘 `Enter`：下载当前选中项
- 右侧下载图标：点击下载当前项
- `Escape`：维持当前清空输入行为

这使列表项可以安全承载“选中后查看更多信息”，而不会再因为点击触发下载。

### 2. Information Hierarchy

每条结果保留两层信息，但重新分配视觉优先级：

- 第一层：文件名
  - 作为主信息
  - 优先分配宽度
  - 选中态时对比度更高
- 第二层：目录路径
  - 作为弱化信息
  - 字号更小、颜色更浅
  - 不再要求在列表中完整展示

同屏扫描时，用户首先识别文件名；目录路径只作为区分上下文的辅助。

### 3. Path Overflow Strategy

列表中的路径不使用跑马灯。

改为两态策略：

- 未选中项优先尝试显示完整路径
  - 仅在当前路径行的真实可用宽度不足时，才退化为中间省略显示
  - 示例：`/work/.../frontend/src/components`
- 宽度判断不依赖固定字符数断点
  - 通过前端测量当前路径行的真实可用宽度
  - 使用与实际文本样式一致的测量方式判断完整路径是否放得下
- 当前选中项不再新增第三块展示区域
  - 第一行文件名取消单行截断，允许换行
  - 第二行路径改为显示完整路径，并允许换行
- 鼠标悬停路径文本时，通过原生 tooltip 或等价轻量提示展示完整路径

这样可以保证：

- 列表密度稳定
- 页面没有持续运动噪音
- 完整路径仍然可访问

### 4. Selected Item Expansion

当前选中项不再通过底部详情栏或额外第三行来展示完整路径。

改为直接复用原有的两行文本区域：

- 第一行：文件名
  - 未选中时保持紧凑单行
  - 选中时允许换行，不再强制截断
- 第二行：路径
  - 未选中时显示中间省略后的缩略路径
  - 选中时改为完整路径，并允许换行

这样完整信息仍然贴着当前结果项本身，不需要用户把视线移动到页面底部，也不会在移动端被输入法遮挡。

### 5. Responsive Behavior

桌面和窄屏统一使用同一套交互模型：

- 较宽屏幕
  - 未选中项仍保持高密度
  - 选中项直接在原位展开两行内容
- 窄屏
  - 同样沿用选中项原位展开
  - 下载图标保留明确点击目标

目标是“同一交互、同一信息位置”，避免桌面和移动端出现不同的信息承载区域。

## Components

### `frontend/src/App.tsx`

负责：

- 重排结果项布局
- 将整行点击从下载改为选中
- 在选中态切换文件名和路径的展示模式
- 接入路径宽度测量结果，决定未选中项是否需要缩略
- 维持下载图标的独立按钮语义

### `frontend/src/hooks/useMeasuredPathDisplay.ts`

负责：

- 测量未选中结果项路径行的真实可用宽度
- 在容器宽度变化时重新计算可展示路径
- 向 `App` 提供“完整显示或中间省略”的展示结果

### `frontend/src/pathDisplay.ts`

继续负责：

- 提取 basename
- 在需要缩略时生成中间省略路径
- 根据测量宽度选择可放下的最优展示字符串

### `frontend/src/hooks/useKeyboardNavigation.ts`

继续负责：

- 上下键切换选中项
- `Enter` 触发下载
- 在结果变化后维护合法选中索引

无需改变核心职责，但要保证“鼠标单击选中”和“键盘选中”保持一致。

### `frontend/src/hooks/useDownload.ts`

继续负责：

- 下载请求
- toast 提示
- 幽灵文件移除

需要移除对“整行点击即下载”的耦合假设，但无需变更下载协议。

## Data Flow

1. WebSocket 返回结果列表
2. `useKeyboardNavigation` 维护当前选中索引
3. `useMeasuredPathDisplay` 根据当前路径行可用宽度和文本测量结果，为未选中项生成展示路径
4. `App` 根据选中索引决定：
   - 哪一行高亮
   - 哪一项取消文件名截断
   - 哪一项始终显示完整路径
   - 哪一项使用按宽度计算后的展示路径
5. 用户单击结果行时，仅更新选中索引
6. 用户点击下载图标或按 `Enter` 时，调用 `handleDownload`

## Accessibility

- 下载图标应是明确的 `button`
- 当前选中项应保留清晰的视觉状态
- Tooltip 只作为补充，不能成为唯一的信息入口
- 键盘路径浏览不依赖鼠标 hover

## Testing

需要覆盖：

- 单击结果行只改变选中项，不触发下载
- 点击下载图标仍会触发下载
- `Enter` 仍下载当前选中项
- 选中项在行内显示完整路径
- 未选中项在宽度足够时显示完整路径
- 未选中项在宽度不足时显示中间省略路径
- 选中项文件名和路径允许换行，不再依赖额外展示区域
- 长路径在列表中使用省略展示，但 tooltip 或选中态可看到完整值
- 容器宽度变化后，未选中项路径展示会重新计算
- 结果被移除时，选中索引仍保持有效

## Risks

- 如果测量逻辑与真实样式不一致，可能导致“明明放得下却仍被缩略”或相反
- 若选中项换行过多，列表滚动位置可能有轻微跳动
- 若下载图标点击区域过小，触屏体验会下降

## Recommendation

首轮实现先采用最稳妥版本：

- 单击选中
- 图标/Enter 下载
- 未选中项基于真实可用宽度决定是否缩略
- 选中项直接在原两行区域内展开完整信息
- tooltip 作为补充

这一版改动集中在前端，信息位置稳定，也更适合移动端和桌面端统一体验。
