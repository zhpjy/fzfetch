# Fzfetch 后端设计文档

## 概述

本文档定义 Fzfetch 后端核心设计，严格遵循当前已经确认的全局架构约束：

- 后端是唯一持有完整文件集合并执行模糊搜索的地方。
- `cache.txt` 只是冷启动加速介质，不是真实数据源。
- 真实数据源始终是单一配置根目录下的实际文件系统。
- 搜索结果允许短时间内存在陈旧数据，文件是否真实存在只在下载时做惰性校验。

本次后端设计已确认的边界如下：

- `cache.txt` 保存绝对路径，每行一个文件。
- 只索引文件，不索引目录。
- 搜索范围与下载沙箱范围使用同一个单一根目录。
- 缓存文件固定放在 `data/cache.txt`，便于后续 Docker volume 持久化。
- 进程启动时必须自动创建 `data/` 和 `data/cache.txt`，如果它们不存在。
- 内存索引只在用户真正使用系统时才装载。
- 内存索引在连续空闲达到可配置超时后自动卸载，默认值为 30 分钟。
- 若缓存距离上次刷新时间超过可配置阈值，则触发后台刷新，默认值为 24 小时。

## 目标

- 在首个真实请求到来之前，保持服务进程本身足够轻。
- 首次搜索请求到来时，通过加载 `data/cache.txt` 立即提供可用搜索能力。
- 刷新行为必须异步执行，不能阻塞正在进行的搜索。
- 通过请求纪元取消机制，避免网络回传和前端渲染洪峰。
- 通过规范化路径校验，保证下载接口的目录沙箱安全。
- 通过后台全盘扫描和下载时惰性校验，实现最终一致性。

## 非目标

- 不使用 `inotify` 等系统级文件监听。
- 不在前端执行任何模糊匹配计算。
- 不在进程启动时常驻完整内存索引。
- 不支持目录浏览或目录级交互语义。
- 本轮不支持多根目录策略。

## 配置模型

后端配置仅包含核心行为所需的最小字段：

- `root_dir: PathBuf`
  唯一允许的文件系统根目录，同时用于搜索扫描与下载校验。
- `data_dir: PathBuf`
  默认值为 `data/`。
- `cache_file: PathBuf`
  默认值为 `data/cache.txt`。
- `refresh_ttl: Duration`
  缓存刷新阈值，默认值为 24 小时。
- `idle_ttl: Duration`
  内存索引空闲卸载阈值，默认值为 30 分钟。
- `cleanup_interval: Duration`
  空闲清理任务的轮询间隔，例如 60 秒即可。
- `top_k: usize`
  搜索返回结果数量上限，默认值为 100。

只支持一个根目录，这样可以让下载沙箱和索引范围保持完全一致，也能避免 `canonicalize` 校验中出现多重前缀判断的歧义。

## 模块划分

后端建议拆分为以下聚焦模块：

- `main.rs`
  负责启动流程：加载配置、确保 `data/` 与 `data/cache.txt` 存在、构建 `AppState`、注册 HTTP 与 WebSocket 路由、启动空闲清理任务。
- `state.rs`
  定义 `AppState`、`IndexManager`、`IndexRuntime`、共享配置与索引刷新广播器。
- `cache.rs`
  负责 `data/cache.txt` 的创建、读取、清洗与原子覆盖写入。
- `scanner.rs`
  负责基于 `walkdir` 递归扫描根目录，并产出 `HashSet` 差异结果。
- `search.rs`
  负责封装 `nucleo`：初始化引擎、批量注入缓存、按差异增删、执行 Top K 搜索。
- `ws.rs`
  负责 WebSocket 协议、请求解析、请求纪元取消，以及向客户端转发 `INDEX_REFRESHED` 广播。
- `api.rs`
  负责安全下载接口和幽灵文件的惰性校验逻辑。

该拆分保证每个文件职责清晰，避免把 HTTP 逻辑、索引生命周期和搜索逻辑揉在一起。

## 运行时状态模型

### AppState

`AppState` 持有以下共享状态：

- 不可变共享配置
- `IndexManager`
- 一个用于广播 `INDEX_REFRESHED` 的通道发送端

### IndexManager

`IndexManager` 是索引相关行为的统一入口，负责：

- 懒加载内存索引
- 判定并调度后台刷新
- 执行空闲卸载
- 为搜索请求提供运行时索引

它通过如下结构持有当前内存索引：

- `RwLock<Option<Arc<IndexRuntime>>>`

其中：

- `None` 表示当前没有内存索引，搜索状态已经被释放。
- `Some(...)` 表示当前存在一份可用的内存索引。

### IndexRuntime

`IndexRuntime` 保存真正驻留内存的搜索态：

- `paths: RwLock<HashSet<String>>`
  表示当前内存中已知的完整文件集合。
- `engine: Mutex<SearchEngine>`
  对 `nucleo` 匹配器和注入器的轻量封装。
- `last_used_at: Mutex<Instant>`
  在每次搜索请求到达时更新，用于空闲卸载判断。
- `last_refresh_at: Mutex<SystemTime>`
  用于判断是否已经超过刷新阈值。
- `refreshing: AtomicBool`
  用于保证同一时刻只有一个后台刷新任务在运行。

搜索引擎与路径集合必须放在同一个运行时对象中，因为刷新时必须同时更新“真值集合”和“匹配引擎”。

`latest_req_id` 不应该存放在 `IndexRuntime` 中。请求纪元取消必须以“每个 WebSocket 连接”为作用域，否则一个客户端发出的新请求会错误地取消另一个客户端正在计算的请求。

## 启动行为

服务启动时只做磁盘准备和服务引导，不加载内存索引：

1. 校验配置中的根目录存在且可用。
2. 若 `data/` 不存在，则创建。
3. 若 `data/cache.txt` 不存在，则创建空文件。
4. 将根目录做一次 `canonicalize`，并把规范化结果保存到共享配置或状态中。
5. 启动 HTTP 路由与 WebSocket 路由。
6. 启动周期性空闲清理任务。

启动路径不应把 `cache.txt` 载入内存，这样才能满足“有用户使用时才装载索引”的要求。

如果 `data/cache.txt` 已存在但为空，启动依然成功。首次真实刷新会在之后把它填充起来。

## 懒加载流程

当首个用户搜索请求到来时，执行以下流程：

1. `IndexManager::ensure_loaded()` 先通过读锁判断当前是否已有运行时索引。
2. 若没有，则进入写路径并读取 `data/cache.txt`。
3. 对每一行做处理，非空行按绝对路径解释。
4. 非法行或不可读行跳过，不因单行问题导致整体失败。
5. 用合法路径构建新的 `HashSet<String>`。
6. 基于该集合初始化新的 `nucleo` 引擎并完成注入。
7. 将 `last_used_at` 设为当前时间。
8. 将 `last_refresh_at` 初始化为缓存文件的修改时间；若无法获取，则设为 UNIX epoch，意味着下一次使用时可立即判定为需要刷新。
9. 将构建好的运行时索引写入 `RwLock<Option<_>>`。

如果缓存文件为空，仍然要创建一份“空索引运行时”。这样首次搜索不会失败，只是返回空结果，同时后续刷新会把真实文件集合补齐。

## 刷新策略

刷新由“使用行为”触发，而不是由“进程启动”触发。

每次搜索请求到来时都执行以下判断：

1. 确保内存索引已存在。
2. 更新 `last_used_at`。
3. 判断 `now - last_refresh_at` 是否超过 `refresh_ttl`。
4. 若已超时，则尝试执行 `refreshing.compare_exchange(false, true, ...)`。
5. 若抢占成功，则启动一个后台异步刷新任务。
6. 若抢占失败，则说明已有刷新在运行，当前请求不再重复发起。

这样可以保证：

- 系统无人使用时不会做无意义扫描。
- 一旦真实用户开始使用，旧缓存会在后台逐步收敛到最新状态。

## 后台刷新流程

后台刷新是最终一致性的核心机制：

1. 用 `walkdir` 递归扫描配置根目录。
2. 只保留元数据表明是普通文件的条目。
3. 将扫描到的每个路径规范化为绝对路径字符串，供搜索展示和下载接口共用。
4. 构建一份新的 `HashSet<String>`，记为 `new_paths`。
5. 读取当前内存中的旧集合，记为 `old_paths`。
6. 计算差异：
   - `added = new_paths - old_paths`
   - `removed = old_paths - new_paths`
7. 通过 `nucleo::Injector` 应用差异：
   - 将 `added` 中的路径逐条注入搜索引擎
   - 将 `removed` 中的路径逐条从搜索引擎中移除
8. 用 `new_paths` 替换内存中的完整文件集合。
9. 将新全集原子覆盖写回 `data/cache.txt`。
10. 更新 `last_refresh_at` 为当前时间。
11. 向所有在线客户端广播 `{"type":"INDEX_REFRESHED"}`。
12. 将 `refreshing` 重置为 `false`。

若扫描失败：

- 保留原有内存索引，不清空。
- 不覆盖 `data/cache.txt`。
- 不发送刷新完成广播。
- 只记录错误并重置 `refreshing`。

若内存索引更新成功，但写回缓存文件失败：

- 内存索引仍保留最新状态。
- 记录错误。
- 后续下一轮刷新仍可修复磁盘缓存。

这里的策略是优先保证活跃用户看到的是尽可能新的内存索引，而不是为了磁盘落盘一致性去阻塞整个流程。

## HashSet 差异规则

刷新中的差异比对严格基于集合运算：

- `added` 包含“扫描结果中存在，但当前内存中不存在”的所有路径。
- `removed` 包含“当前内存中存在，但扫描结果中不存在”的所有路径。

这样可以避免每次刷新都重建整份搜索引擎，也与“使用 `Injector` 做增量更新”的架构要求一致。

由于系统只索引文件，目录级变更不会进入任何差异集合。

## 搜索流程

WebSocket 搜索请求的处理流程如下：

1. 接收 JSON，格式为 `{"req_id": 123, "query": "rust"}`。
2. 解析并校验消息内容。
3. 确保内存索引存在。
4. 更新 `last_used_at`。
5. 将当前请求的 `req_id` 写入“当前 WebSocket 连接独有”的 `AtomicU64`。
6. 如果缓存已过期，则尝试调度后台刷新任务。
7. 基于当前内存中的 `nucleo` 引擎执行模糊匹配。
8. 取 Top 100 结果，每条结果包含 `path` 与 `score`。
9. 搜索完成后再次读取该连接自己的 `AtomicU64`。
10. 如果读取出的值已经不等于本次请求的 `req_id`，说明结果已过期，直接丢弃并返回，不向前端发送任何内容。
11. 如果仍然相等，则发送：
    `{"req_id":123,"data":[{"path":"/a/b.txt","score":99}]}`

这就是必须实现的请求纪元取消逻辑。它可以有效阻止较慢的旧请求在较新的请求之后回传结果，从而避免网络洪峰与前端无效重绘。

## WebSocket 广播行为

每个 WebSocket 客户端连接建立后，都应订阅索引刷新广播通道。

当后台刷新成功完成时，服务端广播：

```json
{"type":"INDEX_REFRESHED"}
```

该消息只作为“后端数据集已变化”的轻量通知，不携带具体 diff。这样可以继续保持“后端权威、前端极简”的架构原则。

## 空闲卸载策略

空闲清理任务按固定时间间隔运行，例如每分钟执行一次：

1. 读取当前运行时索引。
2. 如果当前没有运行时索引，则直接跳过。
3. 比较 `Instant::now()` 与 `last_used_at` 的差值。
4. 若空闲时长小于等于 `idle_ttl`，则保留当前索引。
5. 若空闲时长大于 `idle_ttl` 且当前 `refreshing == false`，则将 `Option` 替换为 `None`，释放整份内存索引。

这一策略的几个重要后果是：

- 被卸载的只是内存搜索索引。
- `data/cache.txt` 永远保留在磁盘上。
- 下次有用户请求时，会重新从缓存文件执行懒加载。

如果当前刷新正在执行，则本轮跳过卸载，避免刷新过程中运行时对象被拆除。

## 缓存文件规则

`data/cache.txt` 采用简单、稳定的文本格式：

- UTF-8 文本文件
- 每行一个绝对文件路径
- 空行忽略

写入时应采用原子替换方式。建议先写入 `data/` 目录下的临时文件，再通过重命名覆盖原 `cache.txt`。

这样可以避免进程崩溃或容器重启时留下半写入状态的损坏文件。

## 下载接口设计

下载接口形式为：

- `GET /download?path=/absolute/path/to/file.pdf`

处理顺序如下：

1. 解析查询参数中的 `path`。
2. 若配置中的根目录还未规范化，则先对其执行 `canonicalize`。
3. 尝试对请求路径执行 `canonicalize`。
4. 如果规范化失败是因为目标已不存在，则返回 `410 Gone`。
5. 如果规范化失败是其他原因，例如路径格式非法，则返回 `400 Bad Request`。
6. 判断规范化后的目标路径是否以前缀形式落在规范化后的根目录之内。
7. 如果不在根目录下，则返回 `403 Forbidden`。
8. 判断目标是否为普通文件。
9. 如果不是普通文件，则返回 `404 Not Found`。
10. 执行惰性校验：
    - 如果在真正返回文件之前发现文件已消失，则返回 `410 Gone`
11. 若目标合法且存在，则以流式方式返回文件内容。

这里最关键的安全点是：前缀判断必须基于规范化后的真实路径，而不能基于原始字符串。这样才能防住 `../../` 一类路径穿越攻击。

## 惰性校验与幽灵文件

搜索结果中允许短时间出现幽灵文件，这是系统设计上有意接受的行为。

原因是：

- 内存索引是最终一致，而不是瞬时强一致。
- 后台刷新是异步的，不会在每个删除动作发生时立刻同步到搜索引擎。

因此，真正的存在性校验点放在下载时：

- 文件存在，则正常流式返回。
- 文件已消失，则返回 `410 Gone`。

前端收到 `410 Gone` 后可以静默移除对应条目；后续某次全量刷新会把它从索引中永久清理掉。

## 错误处理策略

后端错误分为“启动致命错误”和“运行期可恢复错误”。

### 启动致命错误

- 配置中的根目录不存在或不可用
- `data/` 无法创建
- `data/cache.txt` 无法创建

这些错误会破坏系统最基本的持久化或沙箱边界，因此应直接阻止服务启动。

### 运行期可恢复错误

- 缓存文件中存在非法行
- 缓存文件读取部分失败
- WebSocket 搜索请求格式非法
- 后台扫描失败
- 缓存文件重写失败
- 某个 WebSocket 客户端异常断开
- 请求下载的文件已不存在

这些错误应在局部处理，不应导致整个进程崩溃。

## 并发边界

并发控制保持尽量简单直接：

- `RwLock<Option<Arc<IndexRuntime>>>` 控制“当前是否存在内存索引”
- `AtomicBool refreshing` 防止重复启动后台刷新
- 每个 WebSocket 连接各自持有一个 `AtomicU64` 实现请求纪元取消
- 小粒度 `Mutex` / `RwLock` 保护少量可变状态

系统不应在搜索路径上引入全局重写锁。搜索与刷新应尽量并行，刷新只通过有限状态与注入器更新影响当前运行时索引。

## 测试策略

最少应覆盖以下高风险行为：

1. 缓存介质初始化
   - 能创建 `data/`
   - 能创建 `data/cache.txt`

2. 缓存解析
   - 忽略空行
   - 保留绝对路径
   - 遇到非法行不崩溃

3. 差异比对
   - 能正确计算 `added` 与 `removed`

4. 懒加载
   - 能从 `data/cache.txt` 构建运行时索引
   - 缓存为空时能构建空索引

5. 刷新调度
   - 仅在 TTL 过期时触发刷新
   - `refreshing` 能抑制重复刷新

6. 请求纪元取消
   - 旧 `req_id` 的结果会被丢弃

7. 空闲卸载
   - 超过空闲阈值后能释放运行时索引
   - 刷新进行中不会被卸载

8. 下载沙箱
   - 能拦截越界路径
   - 能拒绝非文件目标
   - 幽灵文件会返回 `410 Gone`

## 实现方向

第一版实现应优先保证以下行为正确，而不是过早抽象：

- 懒加载生命周期清晰可预测
- 基于差异的后台刷新行为正确
- 请求纪元取消逻辑正确
- `410 Gone` 惰性校验语义正确
- `canonicalize` 沙箱防线正确

整体实现应保持小而直接，围绕“本地极速文件搜索”这一目标服务，而不是演化成通用分布式索引系统。
