# Mobile Header Compact Design

## Goal

缓解移动端顶部区域在加入语言切换控件后的拥挤问题，保留品牌识别和系统状态可见性，同时避免通过过度缩小控件来牺牲点击体验和可读性。

## Scope

- 只调整前端 header 的信息层级和展示方式
- 只对移动端 `<sm` 做布局取舍，桌面端 `sm+` 尽量保持现状
- 移动端隐藏手动语言切换入口，继续保留自动语言适配逻辑
- 将移动端状态区改为摘要化展示
- 适度收紧 header 的尺寸、间距和留白

不在本次范围内：

- 后端状态协议或 websocket 数据结构调整
- i18n 语言解析逻辑改造
- 新增移动端设置页、抽屉或更多菜单
- 改写桌面端状态区的信息结构

## Problem

当前移动端 header 同时承载三类一级信息：

1. 品牌入口
2. 手动语言切换
3. 状态指示

这三块都放在顶部同一层后，会产生几个直接问题：

- 可用横向空间不足，导致视觉密度过高
- 若简单把三块等比例缩小，会优先伤害点击热区和状态可读性
- 语言切换虽然有用，但在已支持自动语言适配的前提下，不应继续占据移动端一级信息位

## Information Priority

移动端 header 的信息优先级调整为：

1. 品牌
2. 状态
3. 语言切换

对应策略：

- 品牌和状态保留在一级 header
- 语言切换在移动端直接隐藏
- 桌面端继续保留语言切换，以维持现有显式控制能力

## Design

### 1. Header Structure

移动端继续使用单行 header，不改为双行结构。

布局为：

- 左侧：品牌链接
- 右侧：compact 状态指示

这样做的原因是：

- 单行结构对现有页面侵入最小
- 不会把搜索输入框整体继续向下推
- 品牌和状态仍保持首屏即见

### 2. Language Switch Handling

移动端 `<sm` 直接隐藏语言切换控件。

保留以下行为不变：

- 浏览器语言自动解析
- `localStorage` 中已保存语言继续生效
- 桌面端手动切换能力继续保留

这样处理后，移动端不再为低优先级配置项占用一级 header 空间。

### 3. Brand Compaction

品牌区不做激进压缩，只轻缩一档：

- logo 图标尺寸略减
- 标题字号略减
- 品牌区内 `gap` 略减
- header 顶部 `padding` 和底部间距略减

目标是：

- 保留 `FZFETCH` 的识别度
- 给右侧状态区释放少量空间
- 避免 header 因为强压缩而失去主视觉锚点

### 4. Status Summarization

移动端状态区改为 compact pill，只显示一个“主状态”。

移动端 compact 形态保留：

- 状态色或忙碌图标
- 一段主状态文案

移动端 compact 形态移除：

- 工作状态和索引状态的并排双块展示
- `IDLE / SEARCHING / SCANNING` 的独立常驻文字

桌面端仍保留现有完整状态布局，避免移动端摘要化影响桌面信息密度。

### 5. Primary Status Rule

移动端主状态按以下优先级选择：

1. 连接状态：`connecting` / `disconnected` / `error`
2. 索引状态：`pending` / `refreshing` / `unknown`
3. 就绪状态：`index ready`

选择原则是：

- 连接状态和索引状态决定系统是否可正常使用，优先级高于当前工作态
- 工作态是补充信息，在移动端一级 header 中不再单独常驻

### 6. Busy State Presentation

对于进行中状态，compact 状态区显示忙碌反馈，但不再展开完整工作态文案体系。

表现规则：

- `connecting` 或 `refreshing` 等进行中状态显示 loader + 主状态文案
- 非进行中状态显示状态点 + 主状态文案
- 文案保持单行，不扩展成两行，避免 header 再次增高

## Components

### `frontend/src/App.tsx`

负责：

- 在移动端隐藏语言切换入口
- 调整 header 的间距、字号和品牌尺寸
- 保持桌面端 header 结构不变

### `frontend/src/components/StatusIndicator.tsx`

负责：

- 复用现有状态判断逻辑
- 增加移动端 compact 展示
- 为 `sm+` 保留现有完整布局

## Accessibility

- compact 状态区仍需保留清晰的语义标签，确保辅助技术能读出完整状态含义
- 移动端品牌链接点击区域不因轻缩而明显变差
- 不能通过把按钮或 pill 压到过小来换取布局整洁

## Testing

需要覆盖：

- 移动端语言切换入口被隐藏，桌面端入口仍存在
- 移动端 compact 状态只显示主状态
- 连接状态优先于索引状态
- 桌面端完整状态区仍保留工作态与索引态信息
- 不同 locale 下 compact 状态继续使用现有本地化文案

## Risks

- 如果 compact 状态文案过长，仍可能在极窄设备上产生拥挤
- 若实现时把移动端和桌面端状态逻辑混在一起，容易引入回归
- 如果只隐藏语言切换但不同时收紧品牌和 header 间距，改善幅度会有限

## Recommendation

首轮实现采用最保守版本：

- 移动端隐藏语言切换
- 品牌轻缩一档
- 状态区改为单个 compact pill
- 桌面端保持当前完整布局

这条路径改动集中、风险低，并且与当前“品牌优先、状态次之、语言切换降级”的产品取舍一致。
